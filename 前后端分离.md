# 前后端分离前端开发需求

## 1. 项目目标

基于当前 `src/main/resources/static` 中已有的静态页面，实现 Vue 3 前端工程迁移，完成前后端分离。

迁移后的前端项目放在：

```text
frontend
```

本次迁移的核心目标：

1. 保持现有页面布局和视觉样式不变。
2. 保留当前静态页面已有功能。
3. 将原本写在 `static/app.js` 中的逻辑拆分为 Vue 组件、组合式函数、API 层和工具函数。
4. 前端独立构建，生产环境单独部署到 Nginx。
5. 后端只作为 API 服务提供能力。

## 2. 技术栈

前端使用：

1. Vue 3
2. TypeScript
3. Vite
4. Ant Design Vue
5. Axios
6. OpenAPI 自动生成请求代码
7. marked
8. highlight.js
9. DOMPurify

说明：

- 普通 REST API 请求使用 Axios。
- OpenAPI 已由后端配置完成，前端需要基于后端 OpenAPI 文档自动生成接口请求代码。
- SSE / 流式接口不强制使用 Axios，单独封装流式请求工具。
- Ant Design Vue 主要用于基础交互能力，例如提示、弹窗、表单控件等，但整体页面视觉需要保持当前样式。

## 3. Vue 代码规范

所有 Vue 组件统一使用组合式 API：

```vue
<template>
</template>

<script setup lang="ts">
</script>

<style scoped>
</style>
```

要求：

1. 不使用 Options API。
2. 组件逻辑使用 TypeScript。
3. 页面状态优先拆分到 composables 中。
4. API 请求逻辑不直接散落在组件里。
5. 不在组件中硬编码后端完整地址。
6. 样式迁移时优先复用当前 `styles.css` 的视觉规则。

## 4. 页面样式要求

本次不是 UI 改版。

要求保持现有页面风格，包括但不限于：

1. 左侧边栏布局。
2. 主聊天区域布局。
3. 欢迎语位置。
4. 聊天气泡样式。
5. 输入框区域布局。
6. 工具菜单位置。
7. 快速 / 流式模式切换样式。
8. 登录 / 注册弹窗样式。
9. 用户资料面板样式。
10. 用户头像 / 用户资料入口样式。
11. loading 遮罩样式。
12. 通知提示位置和交互体验。

允许：

- 使用 Vue 组件重构 DOM。
- 使用 Ant Design Vue 替代部分基础控件。
- 在不改变视觉结果的前提下优化样式组织。

不允许：

- 改成明显的 Ant Design 后台风格。
- 改变主布局。
- 删除已有用户可见功能。
- 大幅调整颜色、间距、聊天气泡、侧边栏宽度等视觉表现。

## 5. 功能迁移范围

必须保留当前静态页面已有功能。

### 5.1 认证功能

保留：

1. 页面初始化时检查登录状态。
2. 未登录时显示登录弹窗。
3. 登录。
4. 注册。
5. 注册后自动登录。
6. 登出。
7. 登录后显示右上角用户资料入口。
8. 登出后清空当前用户状态和本地聊天缓存。

对应接口：

| 功能 | 方法 | 接口 |
|---|---:|---|
| 当前用户 | GET | `/api/auth/me` |
| 注册 | POST | `/api/auth/register` |
| 登录 | POST | `/api/auth/login` |
| 退出登录 | POST | `/api/auth/logout` |

认证方式：

- 后端使用 `HttpSession`。
- 前端请求必须携带 Cookie。
- Axios 全局配置 `withCredentials: true`。
- 开发环境通过 Vite proxy 代理 `/api`，避免跨域 Cookie 问题。

### 5.2 用户资料功能

保留当前资料面板。

字段包括：

1. 性别。
2. 年龄。
3. 身高 cm。
4. 当前体重 kg。
5. 目标体重 kg。
6. 活动水平。
7. 饮食偏好。
8. 健康备注 / 禁忌。

对应接口：

| 功能 | 方法 | 接口 |
|---|---:|---|
| 获取资料 | GET | `/api/profile` |
| 保存资料 | POST | `/api/profile` |

要求：

1. 点击用户资料入口时打开资料面板。
2. 打开资料面板时加载最新资料。
3. 保存成功后提示用户。
4. 保存失败时显示错误提示。
5. 表单样式保持当前页面风格。

### 5.3 聊天功能

保留两种聊天模式：

1. 快速模式。
2. 流式模式。

快速模式：

| 功能 | 方法 | 接口 |
|---|---:|---|
| 快速聊天 | POST | `/api/chat` |

请求体：

```json
{
  "id": "session_xxx",
  "question": "用户问题"
}
```

流式模式：

| 功能 | 方法 | 接口 |
|---|---:|---|
| 流式聊天 | POST | `/api/chat_stream` |

要求：

1. 输入框为空时禁止发送。
2. 未登录时禁止发送，并打开登录弹窗。
3. 正在流式输出时禁止重复发送。
4. 正在流式输出时禁止切换模式。
5. 用户消息立即显示。
6. assistant 回复支持 Markdown 渲染。
7. 流式回复需要逐步追加显示。
8. 流式完成后将完整 assistant 消息写入当前会话历史。
9. 出错时在聊天区显示错误消息。

### 5.4 Markdown 和代码高亮

迁移后保留：

1. assistant 消息 Markdown 渲染。
2. 流式完成后的 Markdown 渲染。
3. 代码块高亮。
4. AI Ops 报告 Markdown 渲染。

安全要求：

1. Markdown 渲染结果需要经过 XSS 防护。
2. 建议使用 `marked` + `DOMPurify`。
3. 不允许直接将未清洗的 HTML 写入页面。

### 5.5 历史会话

继续使用：

```text
localStorage + 后端会话
```

策略保留当前设计：

1. 登录后从后端加载会话摘要列表。
2. 本地 `localStorage` 缓存会话详情。
3. 后端会话列表与本地缓存进行合并。
4. 点击历史会话时：
   - 如果本地已有消息详情，直接展示。
   - 如果本地无详情，请求后端会话详情。
5. 删除历史会话时：
   - 前端先更新本地列表。
   - 同步请求后端删除接口。
   - 后端删除失败时提示用户。
6. 本地最多缓存 50 条。
7. 本地历史保留 30 天。

对应接口：

| 功能 | 方法 | 接口 |
|---|---:|---|
| 会话列表 | GET | `/api/chat/sessions` |
| 会话详情 | GET | `/api/chat/session/{sessionId}` |
| 删除会话 | POST | `/api/chat/clear` |

### 5.6 新建对话

保留当前行为：

1. 当前正在输出时禁止新建对话。
2. 当前会话有内容时保存或更新当前会话。
3. 清空当前消息列表。
4. 生成新的 sessionId。
5. 默认切换回快速模式。
6. 恢复欢迎页居中状态。
7. 更新左侧历史会话列表。

### 5.7 文件上传

保留当前文件上传能力。

要求：

1. 只有管理员可以看到上传入口。
2. 非管理员不显示上传入口。
3. 支持文件类型：
   - `.txt`
   - `.md`
   - `.markdown`
4. 前端限制文件最大 50MB。
5. 上传过程中显示 loading 遮罩。
6. 上传完成后在聊天区域显示成功消息。
7. 上传失败时显示错误提示。

对应接口：

| 功能 | 方法 | 接口 |
|---|---:|---|
| 上传文档 | POST | `/api/upload` |

请求类型：

```text
multipart/form-data
```

字段：

```text
file
```

### 5.8 AI Ops 功能

AI Ops 需要保留，但设计为可插拔模块。

要求：

1. 保留当前右上角 AI Ops 按钮。
2. 点击后自动新建对话。
3. 显示“分析中...”消息。
4. 调用 `/api/ai_ops`。
5. 以 SSE 流式方式展示分析过程和最终结果。
6. 支持 Markdown 渲染。
7. 出错时在当前 AI Ops 消息中展示错误内容。

对应接口：

| 功能 | 方法 | 接口 |
|---|---:|---|
| AI Ops 流式分析 | POST | `/api/ai_ops` |

可插拔要求：

1. AI Ops 相关代码需要独立拆分，不与普通聊天强耦合。
2. 建议拆分为独立组件和组合式函数，例如：
   - `AIOpsButton.vue`
   - `useAIOps.ts`
   - `aiOpsStream.ts`
3. 如果后续不想显示 AI Ops，只需要注释或移除少量入口代码即可。
4. AI Ops 模块不应影响普通聊天、登录、历史会话、资料面板等核心功能。
5. 可以通过配置开关控制是否显示，例如：

```ts
export const featureFlags = {
  aiOps: true
}
```

页面中根据开关控制：

```vue
<AIOpsButton v-if="featureFlags.aiOps" />
```

这样以后如果不想显示 AI Ops，可以把配置改为：

```ts
export const featureFlags = {
  aiOps: false
}
```

或者直接注释掉入口组件。

## 6. SSE / 流式请求处理要求

当前有两个流式接口：

1. `/api/chat_stream`
2. `/api/ai_ops`

这两个接口需要单独封装，不建议依赖 OpenAPI 生成代码直接处理。

建议前端新增：

```text
src/api/stream.ts
```

职责：

1. 发起 POST 请求。
2. 读取 `ReadableStream`。
3. 解析 SSE 行格式。
4. 支持 `event:` 行。
5. 支持 `data:` 行。
6. 支持 JSON 格式的 `SseMessage`。
7. 支持 `content`、`done`、`error` 类型。
8. 对外通过回调返回增量内容、完成事件和错误事件。

普通 REST API 使用 Axios；流式接口使用 `fetch`。

## 7. OpenAPI 生成要求

后端已经配置 OpenAPI，因此前端需要：

1. 在 `frontend` 中配置 OpenAPI 代码生成脚本。
2. 生成代码放在独立目录。
3. 生成代码不手动修改。
4. 业务层不要直接散落使用底层请求细节。
5. 如果生成工具不适合处理 SSE 接口，SSE 接口单独手写封装。

建议目录：

```text
frontend/src/api/generated
```

建议保留手写封装：

```text
frontend/src/api/client.ts
frontend/src/api/stream.ts
frontend/src/api/auth.ts
frontend/src/api/chat.ts
frontend/src/api/profile.ts
frontend/src/api/upload.ts
frontend/src/api/aiOps.ts
```

其中：

- `generated` 放自动生成代码。
- `client.ts` 放 Axios 实例。
- `stream.ts` 放 SSE 流式请求。
- 业务 API 文件可以对生成代码做二次封装。

## 8. 环境区分

前端需要支持开发和生产环境。

建议文件：

```text
frontend/.env.development
frontend/.env.production
```

开发环境：

```env
VITE_API_BASE_URL=/api
VITE_ENABLE_AI_OPS=true
```

生产环境：

```env
VITE_API_BASE_URL=/api
VITE_ENABLE_AI_OPS=true
```

开发环境通过 Vite proxy 转发：

```text
/api -> http://localhost:9900
```

生产环境通过 Nginx 转发：

```text
/api -> 后端 Spring Boot 服务
```

## 9. Nginx 部署要求

生产环境前端单独部署到 Nginx。

建议部署结构：

```text
frontend dist -> Nginx html root
/api -> reverse proxy -> Spring Boot backend
```

Nginx 需要支持：

1. SPA history fallback。
2. `/api` 反向代理到后端。
3. Web 流式响应不被缓冲影响。
4. 上传文件大小限制不小于 50MB。
5. Cookie 正常透传。

需要注意 SSE 接口：

- `/api/chat_stream`
- `/api/ai_ops`

Nginx 对这些接口建议关闭代理缓冲：

```nginx
proxy_buffering off;
```

上传大小建议：

```nginx
client_max_body_size 50m;
```

## 13. 状态管理建议

当前项目是单页应用，建议先使用 composables，不强制引入 Pinia。

建议状态拆分：

| composable | 职责 |
|---|---|
| `useAuth.ts` | 登录、注册、登出、当前用户 |
| `useProfile.ts` | 资料加载、资料保存 |
| `useChat.ts` | 当前会话、发送消息、流式消息 |
| `useChatHistory.ts` | localStorage 和后端历史合并 |
| `useUpload.ts` | 文件校验、上传 |
| `useAIOps.ts` | AI Ops 流式请求和状态 |
| `useMarkdown.ts` 或 `utils/markdown.ts` | Markdown 渲染 |

如果后续前端页面明显增多，可以再引入 Pinia。

11. 历史会话列表正常。
12. 历史会话详情加载正常。
13. 删除会话正常。
14. localStorage + 后端会话混合逻辑正常。

## 15. 最终实现方向

采用以下方案：

1. Vue 前端独立工程：`frontend`。
2. 样式保持当前 `static` 页面，不做 UI 改版。
3. OpenAPI 用于普通接口生成：认证、资料、聊天普通接口、会话、上传等。
4. SSE 手写 stream client：聊天流式和 AI Ops 流式。
5. AI Ops 可插拔：单独组件、单独 composable、配置开关控制显示。
6. 历史会话维持现状：`localStorage + 后端会话`。
7. 生产部署到 Nginx：前端静态资源由 Nginx 托管，`/api` 反代后端。
8. 不做 Milvus 页面。
